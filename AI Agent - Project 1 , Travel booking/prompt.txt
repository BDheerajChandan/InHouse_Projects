# AI Travel Booking Agent — Complete Project Reference Prompt

This is the **master prompt** to give to any AI assistant (Claude, GPT-4, etc.) when working on this project. Paste this entire document at the start of a new conversation to give the AI full context.

## 🧭 Project Overview

A full-stack AI-powered travel booking system.

- **Frontend**: React + Vite chat interface
- **Backend**: FastAPI + PostgreSQL
- **AI Layer**: OpenAI GPT or Google Gemini — understands natural language, picks the right tool, executes PostgreSQL operations
- **No raw SQL from the user** — the LLM generates structured JSON → Python tools execute the SQL

### What it can do (by natural language):
| User says | Action |
|---|---|
| "Book a train from BBSR to VSKP on 25 March 2026" | INSERT into `train` + `travel_history` |
| "Book bus and flight from HWH to BBSR on 27 March" | INSERT into `bus`, `flight`, `travel_history` |
| "Book bus HWH→BBSR on 27 March, return car on 3 April" | INSERT 2 rows with different routes/dates |
| "Show me all travel history" | SELECT all from `travel_history` |
| "Show me train bookings between Jan and June" | SELECT with date filter |
| "Update train from BBSR to VSKP on 25 March to 27 March" | UPDATE by route match (no ID needed) |
| "Change bus date to 29 March and departure to VSKP" | UPDATE multiple fields at once |
| "Update booking #3 date to 27 April" | UPDATE by explicit ID |
| "Cancel booking #5" | DELETE from history + vehicle table |
| "Hi / Good morning / Help" | Friendly human response, no LLM cost |

**Critical rule**: `config.py` lowercases `DB_NAME` and `TABLE_NAME` at load time. PostgreSQL stores all unquoted identifiers in lowercase. Never use uppercase table names in SQL.

## 🗄️ PostgreSQL Schema

**Database**: `booking` (auto-created on startup)

**Tables created automatically** (all identical schema):
- `train`, `bus`, `flight`, `car`, `bike` — vehicle-specific tables
- `travel_history` — combined mirror of all vehicle tables (from `TABLE_NAME` in .env)

**Column schema** (every table):
```sql
id          SERIAL       PRIMARY KEY
vehicle     VARCHAR(20)  NOT NULL          -- train / bus / flight / car / bike
from_loc    TEXT         NOT NULL          -- departure city (full name, title case)
to_loc      TEXT         NOT NULL          -- arrival city (full name, title case)
travel_date DATE                           -- YYYY-MM-DD (nullable)
booked_at   TIMESTAMP    DEFAULT NOW()     -- auto-set
status      VARCHAR(20)  DEFAULT 'confirmed'
```

**Important**: Every booking INSERT goes into BOTH the vehicle table AND `travel_history`.

## 🤖 Agent Architecture & Data Flow

```
User types message in React chat
         ↓
POST /api/chat  { "message": "..." }
         ↓
routers/chat.py  →  run_agent(message)
         ↓
booking_agent.py:
  Step 0: Is it a greeting? → instant human reply (no LLM cost)
  Step 1: Call LLM (OpenAI or Gemini)
         - Sends compact system prompt + user message
         - max_tokens=250 for fast JSON response
  Step 2: Extract JSON from response (brace-counting, no regex)
  Step 3: Expand city abbreviations (safety net: hwh→Howrah, bbsr→Bhubaneswar etc.)
  Step 4: dispatch_tool(tool_name, arguments) → tools.py
         ↓
tools.py:
  - create_booking: INSERT into vehicle table + travel_history
  - create_multi_booking: multiple bookings, different routes/dates
  - get_travel_history: SELECT with dynamic WHERE filters
  - update_booking_by_id: UPDATE by explicit #ID
  - update_booking_by_query: UPDATE by route/date match, supports multiple fields
  - delete_booking: DELETE by ID
         ↓
db.py: execute_query() / execute_write() → PostgreSQL
         ↓
JSON response → frontend renders booking card / history table
```

## 🛠️ LLM Tools Reference

### 1. `create_booking`
```json
{
  "tool": "create_booking",
  "arguments": {
    "vehicles": ["train"],
    "source": "Bhubaneswar",
    "destination": "Visakhapatnam",
    "date": "2026-03-25"
  }
}
```
- `vehicles` is a **list** — single `["train"]` or multi `["bus","car"]` for same route
- `date` can be `null` if not mentioned
- Inserts into each vehicle table AND `travel_history`

### 2. `create_multi_booking`
```json
{
  "tool": "create_multi_booking",
  "arguments": {
    "bookings": [
      {"vehicle": "bus",  "source": "Howrah",       "destination": "Bhubaneswar", "date": "2026-03-27"},
      {"vehicle": "car",  "source": "Bhubaneswar",  "destination": "Howrah",      "date": "2026-04-03"}
    ]
  }
}
```
- Use when vehicles have **different routes OR different dates**
- Each booking gets its own vehicle table insert + travel_history insert

### 3. `get_travel_history`
```json
{
  "tool": "get_travel_history",
  "arguments": {
    "vehicle": "train",
    "start_date": "2026-01-01",
    "end_date": "2026-06-30"
  }
}
```
- All args are optional/nullable
- Queries `travel_history` table (TABLE_NAME from .env)

### 4. `update_booking_by_id`
```json
{
  "tool": "update_booking_by_id",
  "arguments": {
    "booking_id": 3,
    "vehicle": "train",
    "field": "travel_date",
    "value": "2026-03-27"
  }
}
```
- Use when user gives an explicit booking **#ID**
- `field` options: `"from"` | `"to"` | `"travel_date"` | `"status"`

### 5. `update_booking_by_query` ← KEY TOOL
```json
{
  "tool": "update_booking_by_query",
  "arguments": {
    "vehicle": "bus",
    "source": "Howrah",
    "destination": "Bhubaneswar",
    "current_date": "2026-03-27",
    "updates": [
      {"field": "travel_date", "value": "2026-03-29"},
      {"field": "from",        "value": "Visakhapatnam"}
    ]
  }
}
```
- **`source` / `destination` / `current_date`** = FILTERS to FIND the existing booking
- **`updates`** = list of changes to apply (can be multiple fields at once)
- Falls back to old `field` + `value` single args for backward compatibility

### 6. `delete_booking`
```json
{
  "tool": "delete_booking",
  "arguments": {
    "booking_id": 5,
    "vehicle": "train"
  }
}
```

## 🐛 Known Issues & Fixes Applied

### 1. `relation "travel_history" does not exist`
**Cause**: PostgreSQL lowercases identifiers. `Travel_history` ≠ `travel_history`.
**Fix**: `config.py` lowercases `TABLE_NAME` and `DB_NAME` at load time. All SQL uses lowercase directly.

### 2. Frontend shows `—` for From/To columns
**Cause**: DB returns `from_loc`/`to_loc` but frontend reads `from`/`to`.
**Fix**: `_serialize()` in tools.py adds both keys: `from_loc` + alias `from`, `to_loc` + alias `to`.

### 3. Update query fails — city abbreviation not expanded
**Cause**: LLM sends `"Hwh"` but DB has `"Howrah"`.
**Fix**: `_expand_cities_in_args()` runs after LLM response, expands all city fields including `source`, `destination`, and values in `updates` list.

### 4. Multi-booking with different dates only used one date
**Cause**: `create_booking` takes one date for all vehicles.
**Fix**: New `create_multi_booking` tool — each booking has its own vehicle, source, destination, date.

### 5. Cannot update multiple fields in one message
**Cause**: Old `update_booking_by_query` only took one `field` + `value`.
**Fix**: New `updates: list` parameter — `[{field, value}, {field, value}]` — all applied to matched rows.

### 6. OpenAI `TypeError: proxies` / timeout
**Cause**: `openai==1.30.1` + `httpx>0.26.0` conflict. Also large system prompt = slow.
**Fix**: 
- `requirements.txt` pins `httpx==0.26.0`
- `booking_agent.py` uses explicit `httpx.Client` (Mode 1) + raw httpx fallback (Mode 2)
- System prompt < 300 tokens, `max_tokens=250` on response
- Frontend axios timeout = 90 seconds with AbortController cancel

## 🎨 Frontend Architecture

### Design System
- **Font**: Outfit (UI) + JetBrains Mono (code/mono) — Google Fonts
- **Theme**: Deep navy dark theme with blue accent (`#388bfd`)
- **CSS**: Plain CSS with custom properties (no Tailwind, no CSS-in-JS)
- **Layout**: Sidebar + Chat area, responsive (sidebar collapses on mobile)

### State Management
All state lives in `useChat()` hook (`chatStore.js`):
- `messages` — conversation history
- `loading` — LLM request in progress
- `backendOk` — health check result
- `send(text)` — send message with AbortController
- `cancel()` — abort in-flight request
- `retry()` — resend last message
- `clearChat()` — reset conversation

### API Layer (`services/api.js`)
- Axios instance, `baseURL` from `VITE_API_BASE_URL` env var
- `chatWithAgent(message, signal)` — POST /api/chat with AbortController support
- `healthCheck()` — GET /health (5s timeout)
- Human-readable error messages for timeout, network, server errors

### Message Rendering (`MessageBubble.jsx`)
- Reads both `from_loc`/`from` and `to_loc`/`to` — handles both DB column names and aliases
- `BookingCard` — visual route card for single bookings
- `MultiBookingCards` — stacked cards for multi-vehicle bookings
- `HistoryTable` — sortable table with status badges
- `InlineMarkdown` — renders `**bold**`, `*italic*`, `- list` items

## 🔑 Key Design Decisions

| Decision | Reason |
|---|---|
| `config.py` lowercases TABLE_NAME and DB_NAME | PostgreSQL case sensitivity — prevents "relation does not exist" |
| `_serialize()` adds `from`/`to` aliases | Frontend and DB use different column name conventions |
| `_expand_cities_in_args()` after LLM | LLM sometimes skips abbreviation expansion — Python is the safety net |
| `max_tokens=250` on LLM response | JSON tool call is < 150 tokens — drastically reduces latency |
| `httpx==0.26.0` pin | `openai==1.30.1` incompatible with newer httpx |
| AbortController in frontend | User can cancel slow LLM requests instead of waiting for timeout |
| Greeting detection without LLM | `hi/hello/thanks/bye` handled by Python set lookup — zero API cost |
| `updates: list` in update_booking_by_query | Single request can update multiple fields (date + location together) |
| `create_multi_booking` separate tool | Different routes/dates per vehicle can't fit in single `create_booking` |

## 🔧 Debugging Guide

### Backend terminal shows everything:
```
[Config] — settings loaded from .env
[DB]     — database/table operations
[Agent]  — message received, tool selected, city expansions
[LLM]    — OpenAI/Gemini call, token count, mode (SDK vs raw httpx)
[SQL]    — every SQL statement and params before execution
[Tools]  — tool function entry/exit
[Dispatch] — tool routing
[Router] — HTTP request/response
```

### Common errors and fixes:

| Error | Cause | Fix |
|---|---|---|
| `relation "X" does not exist` | Mixed case table name in SQL | config.py lowercases TABLE_NAME |
| `LLM error: proxies` | httpx version conflict | `httpx==0.26.0` in requirements.txt |
| `timeout exceeded` | Prompt too long / slow API | max_tokens=250, compact prompt |
| `No matching booking found` | City abbreviation not expanded | Check CITY_MAP in booking_agent.py |
| `Frontend shows — in From/To` | Reading wrong column name | _serialize() adds from/to aliases |
| `CORS error` | Frontend URL not in ALLOWED_ORIGINS | Add to .env ALLOWED_ORIGINS |
